1. Supported Upload Formats

The system accepts firewall policy exports in four file formats:

.xlsx — Excel Workbook

Uses the first worksheet. Expected columns: source, destination, protocol, port, action, log. Header names are flexible (case-insensitive, common aliases accepted).

.csv — Comma-Separated Values

Auto-detects schema from headers. Supports both generic firewall rule CSVs and vendor-specific Palo Alto exports with zone/application columns. UTF-8 encoding expected.

.json — JSON Policy Export

Juniper SRX JSON format. Must contain the security policy hierarchy with from-zone, to-zone, match criteria, and actions.

.xml — XML Policy Export

Juniper SRX XML format. Standard <security><policies> hierarchy as returned by show security policies | display xml.

Tip: The header row is required for CSV/XLSX. Files without a recognisable header will be rejected with a 400 error.

2. Supported Vendors

Palo Alto Networks

  • CSV export from Panorama or NGFW
  • Auto-detected via column headers (Source Zone, Destination Zone, Application, Service, etc.)
  • Multi-value fields (e.g. multiple zones) — first token is used
  • Application-default service mapping supported

Juniper SRX

  • JSON export (show security policies | display json)
  • XML export (show security policies | display xml)
  • Supports from-zone/to-zone, match addresses, applications, actions
  • Nested policy structures automatically flattened

Generic / Other Vendors

  • CSV or XLSX with standard columns
  • Required: source, destination, protocol, port, action
  • Optional: log, rule_name, source_interface, destination_interface
  • Works for Check Point, Fortinet, Cisco ACL exports (when formatted as CSV)

3. How to Upload

  1. Go to the Upload page.
  2. Enter your API Key (provided by your administrator).
  3. Select the firewall policy file (.xlsx, .csv, .json, or .xml).
  4. Choose which compliance standards to evaluate against (ISO 27001, CIS v8.1, PCI-DSS).
  5. Optionally enable "Download cleaned artifact" for a normalised version of your data (CSV or JSON).
  6. Click Generate Report.
  7. The HTML compliance report downloads automatically. If cleaned artifact is enabled, that also downloads.
Tip: Maximum upload size is 5 MB. For very large rulesets, split into multiple files (max 500 rules per file recommended).

4. Compliance Standards

Each rule is evaluated against the selected standards. Available frameworks:

StandardScopeKey Controls Checked
ISO 27001 Information security management Logging enabled, deny-by-default, no overly permissive rules, source/destination specificity
CIS v8.1 Critical security controls Protocol restrictions, port hygiene, action validation, network segmentation indicators
PCI-DSS Payment card data protection Explicit deny rules, logging on all permit rules, restricted high-risk ports, no "any/any" rules

5. Report Layout

The downloaded HTML report contains these sections:

Header & Metadata

Shows generation timestamp, detected schema type, and overall compliance status (COMPLIANT / NON-COMPLIANT).

Summary Statistics

MetricMeaning
Total rules evaluatedNumber of valid rules processed from your file
AcceptableRules that passed all selected compliance checks
Requires remediationRules that failed one or more controls
Invalid rowsRows that couldn't be parsed (bad format, missing fields)
Overall statusCOMPLIANT if zero denied + zero invalid; NON-COMPLIANT otherwise

Failed Controls Table

Lists each control that was violated and how many rules triggered it. Use this to prioritise remediation — fix the most frequent control failures first.

Failed Standards Table

Shows which standards had the most violations, helping you understand which compliance programme has the biggest gaps.

Per-Rule Findings

Every evaluated rule gets its own section showing:

  • Row number (matches your original file)
  • Verdict: ACCEPTABLE or REQUIRES REMEDIATION
  • Risk level: LOW, MEDIUM, HIGH, or CRITICAL
  • Target: source → destination protocol/port
  • Reason: Why it passed or failed
  • Controls failed: Specific control IDs that triggered
  • Standards failed: Which frameworks the rule violates

6. Reading the Results

Verdicts

VerdictMeaningAction Required
ACCEPTABLE Rule passes all selected compliance checks None — rule is compliant
REQUIRES REMEDIATION Rule fails one or more controls Review the listed controls and modify the rule accordingly

Common Failure Reasons

  • Logging not enabled: Rule has log set to "none" or missing — enable logging on all permit rules
  • Overly permissive source/destination: Using "any" — restrict to specific addresses or ranges
  • High-risk port without restriction: Sensitive ports (e.g. 23/telnet, 21/ftp) should be denied or tightly scoped
  • Missing explicit deny: Rulesets should end with a deny-all to enforce least-privilege

8. Cleaned Artifacts

When you enable "Download cleaned artifact", the system produces a normalised version of your rules:

  • CSV format: Standard columns (source, destination, protocol, port, action, log) regardless of input vendor format
  • JSON format: Array of normalised rule objects with consistent field names

Invalid rows are excluded from the cleaned output. Use this artifact to import rules into other systems or compare normalised data across vendors.
